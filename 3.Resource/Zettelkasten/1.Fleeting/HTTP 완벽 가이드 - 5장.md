### 웹 서버란?
- 월드 와이드 웹의 일부며, HTTP 요청을 처리하여 응답을 반환한다.
- 웹 서버는 물리 서버와 소프트웨어를 모두 포괄하며, 자신의 리소스를 관리하거나 임의의 기능을 관리하기 위한 기능을 제공할 수 있다.
- 웹 서버는 커넥션을 맺고, 요청을 수신하여 이를 처리한 결과를 응답하는 HTTP 트랜잭션 처리 및 필요시 해당 결과를 로깅할 수 있다.

### ident 프로토콜이란?
- 웹 서버가 로깅 등의 목적을 위해 클라이언트의 정보를 얻고자 하는 경우, `ident` 프로토콜을 통해 필요한 정보를 획득할 수 있다.
- 그러나 이는 클라이언트가 해당 프로토콜을 지원해야 하며, HTTP 트랜잭션을 지연시키기에 잘 사용되지 않는다.

### Docroot란?
- 웹 서버의 파일 시스템에 위치한 임의의 폴더를 웹 서버를 위해 예약한 경우, 해당 폴더를 문서 루트로 지칭하게 된다.
- 웹 서버는 요청 받은 리소스를 해당 문서 루트로부터 찾아 응답하게 되며, 내부 구현을 드러내지 않기 위해 상대 경로를 포함하는 요청 처리에 주의해야 한다.

### 동적 컨텐츠 리소스 매핑
- 웹 서버는 요청에 맞는 리소스를 생성하는 애플리케이션 서버에 URI를 매핑할 수 있으며, 이를 동적 컨텐츠 리소스 매핑이라는 용어로 지칭한다.
- 예를 들어, 웹 애플리케이션 서버는 웹 서버와 복잡한 서버 애플리케이션을 연결한다.
- 최근의 애플리케이션 서버는 Java 서블릿과 같이 강력한 서버측 동적 컨텐츠 생성 기능을 지원한다.

### MIME 결정 방식
- 서버가 리소스를 식별한 후, 응답에 `Content-Type`을 포함해야 한다면 리소스의 MIME 타입을 식별할 수 있어야 한다.
- 예를 들어, 파일의 확장자를 검사하거나 파일의 내용을 직접 검사하여 패턴을 명시하는 매직 파일로부터 적절한 패턴을 찾아 MIME 타입을 결정할 수 있다.

### 프록시란? 
- 프록시는 클라이언트와 웹 서버 사이에 위치하여 여러 기능이나 혜택을 제공한다.
- 일반적으로 중개 위치에서 프로토콜을 변경하면 게이트웨이, 그 외의 경우 프록시라고 지칭하지만 최근에는 이러한 차이가 모호한 경우가 많다.

### 프록시와 URI의 형태
- 클라이언트가 명시적인 프록시를 사용하는 경우 전체 URI를 사용하고, 웹 서버와 직접 소통하는 경우 부분 URI를 사용한다.
	- 이를 초기 HTTP 서버들의 구현에 따른 어쩔 수 없는 선택이다.
- 반면, 리버스 프록시 등과 같이 클라이언트가 프록시의 존재를 몰라 부분 URI를 전달한 경우 프록시는 `Host` 헤더의 내용을 토대로 대상 서버를 식별할 수 있어야 한다.
	- 완전 URI가 요청된 경우라면 프록시가 이를 그대로 활용할 수 있으나, 부분 URI 외에 어떠한 정보도 존재하지 않는다면 에러를 반환해야 한다.
