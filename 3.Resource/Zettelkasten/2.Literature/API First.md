첫번째 글을 부분부분 번역하였음
```
API(애플리케이션 프로그래밍 인터페이스)는 현대 컴퓨팅만큼이나 오래 전부터 존재해 왔습니다. 수십 년 전에 서로 다른 소프트웨어 애플리케이션이 통신할 수 있는 수단으로 등장했습니다. 그리고 오늘날에도 여전히 그 역할을 수행하며 컴퓨터, 휴대폰, 스마트 기기가 서로 연결될 때 백그라운드에서 보이지 않게 작동합니다.

하지만 API는 단순한 인터페이스의 역할을 넘어 진화해 왔습니다. 지난 10년 동안 API는 최신 소프트웨어와 비즈니스의 기본 구성 요소가 되었습니다. Amazon.com, Netflix와 같은 기술 선구자부터 100년 전통의 식료품 체인점, 연방 기관에 이르기까지 조직은 API를 사용하여 외부에 새로운 서비스를 제공하고 내부적으로 효율성을 높이고 있습니다.

API의 성장은 새로운 현실을 반영합니다: 기술 사용자는 여러 디바이스에 걸친 경험을 요구합니다. 이들은 데이터와 서비스를 여러 플랫폼에서 즉시 사용할 수 있고 공유할 수 있기를 기대합니다. 이는 외부 고객에게 서비스를 제공하든 내부 직원에게 서비스를 제공하든 모든 비즈니스가 사실상 소프트웨어 비즈니스라는 것을 의미합니다.
```

```
API First?
API 우선 접근 방식(API first)은 소프트웨어 개발 프로세스의 시작 단계에서 API의 우선순위를 정하여 API를 소프트웨어의 구성 요소로 포지셔닝합니다. API 우선 조직은 API를 나중에 고려하는 대신 다른 코드를 작성하기 전에 API를 먼저 개발합니다. 이를 통해 팀은 API를 통해 제공되는 내부 및 외부 서비스로 애플리케이션을 구성할 수 있습니다.

API first는 어떻게 동작할까?
조직이 API 우선 개발 모델을 채택하려면 API의 우선순위를 정하고, 조직에서 퍼블릭, 프라이빗, 파트너 API의 역할을 인식하고, API 수명주기와 API 우선 개발이 되기 위해 필요한 도구를 이해해야 합니다.
```

```
API 우선 순위 지정하기
가장 선견지명이 있는 기업들은 소프트웨어 개발에 API 우선 접근 방식을 취합니다. 개발자는 한 줄의 코드를 작성하기 전에 비즈니스와 협력하여 먼저 API를 설계하거나 구축합니다. 이를 통해 기본 앱이 내부 및 외부 애플리케이션과 원활하게 연결될 수 있습니다. 이렇게 하면 앱의 기능이 확장되고 파트너와 최종 사용자가 앱에 액세스할 수 있습니다.

API 우선주의는 애플리케이션을 지원하는 API의 우선순위를 정하고, 단일 애플리케이션을 제공하기 위해 허둥대다가 나중에 API를 만드는 것이 아니라 애플리케이션이 비즈니스에 제공할 수 있는 가치에 집중하는 것을 의미합니다. 이러한 미래 지향적인 접근 방식을 통해 API를 통해 비즈니스의 여러 부서에서 애플리케이션을 여러 용도로 채택할 수 있습니다.

API는 한 번에 끝나는 프로젝트가 아닙니다. API는 유지 관리 및 개선이 필요한 핵심 구성 요소입니다. 기업들은 이를 인식하고 이를 지원하기 위한 팀을 구성하고 있습니다.
```

```
Private, partner, public API들
사람들이 API를 떠올릴 때 가장 먼저 떠올리는 것은 eBay나 Stripe 같은 회사의 공개 API입니다. 이러한 API는 수백만 개의 중소기업의 성장을 도왔고 수천억 달러에 달하는 강력한 기술 플랫폼을 만들어냈습니다.

하지만 소프트웨어 개발자들이 가장 많은 노력을 기울이는 곳은 비공개 또는 내부 API입니다. 실제로 37,000명 이상의 API 전문가를 대상으로 설문조사를 실시한 Postman의 2022 API 현황 보고서에 따르면 개발자가 사용하는 API의 58%는 내부용으로만 사용됩니다.

비공개 API를 사용하면 클라우드에 저장된 다양한 애플리케이션이 데이터와 서비스를 공유하고 실행 가능한 인사이트를 제공할 수 있습니다. 이 작업의 대부분은 자동으로 수행될 수 있으므로 모든 직급의 직원이 조직 전체에 대한 가시성을 확보할 수 있습니다.

한 번 구축된 비공개 API는 비즈니스 전반에 걸쳐 재사용할 수 있으므로 더 빠르게 제공하고 귀중한 개발자 리소스를 절약할 수 있습니다.

조직의 요구가 확장되고 비즈니스 파트너와의 상호 작용이 필요해짐에 따라 파트너 API가 등장합니다. 파트너 API를 통해 조직은 엄선된 사용자 및 고객과 API를 공유하여 협업, 비즈니스 파트너십 구축, 타겟 피드백 수집의 기회를 제공할 수 있습니다. API 현황 보고서에 따르면 파트너 API는 조직이 보유한 API의 27%를 차지합니다.

API 현황 보고서에 따르면 웹에서 누구나 공개적으로 사용할 수 있는 API인 퍼블릭 API는 조직이 보유한 API의 약 15%를 차지합니다.

그렇다면 조직은 API에 얼마나 많은 노력을 기울이고 있을까요? 개발자의 약 51%는 조직 개발 노력의 절반 이상을 API에 투자하고 있다고 답했습니다.
```

```
API 수명주기
API 플랫폼에서 운영의 이점을 최대한 활용하고 여러 팀에 걸쳐 수백, 수천 개의 API를 효과적으로 관리하려면 잘 정의된 API 수명 주기가 필수적입니다.

조직 전체에서 API 수명 주기에 대한 이해를 공유하고 이를 설명하는 공통 어휘를 보유하면 팀원들이 동일한 정보를 공유하여 기업 운영에 필요한 생산성, 품질 및 거버넌스를 갖춘 API를 개발하는 데 도움이 됩니다.

Postman의 전 세계 사용자들에게서 가장 흔히 볼 수 있는 단계를 기준으로 API 수명 주기를 8단계로 나누었습니다. API의 유형과 신규인지 기존인지에 따라 팀의 라이프사이클 진입 지점이 달라질 수 있습니다.
```

```
왜 API first인가?
API 우선 개발 모델을 채택하면 개발자와 조직 모두에게 상당한 이점이 있습니다.

API first의 가치 - 개발자
API 우선 접근 방식은 더 강력하고 탄력적인 소프트웨어를 제작할 뿐만 아니라 더 짧은 시간 내에 소프트웨어를 제작할 수 있습니다. 개발자의 작업이 더 쉬워져 병렬로 작업하고 다른 사람의 코드를 디버깅하는 데 소요되는 시간을 줄일 수 있습니다.

개발자는 기존 소프트웨어를 재창조하는 대신 혁신에 집중할 수 있습니다. 또한 API를 사용하면 원하는 기술, 플랫폼, 프로그래밍 언어를 선택할 수 있습니다.

이는 궁극적으로 API를 우선시하는 기업의 개발자의 만족도가 높다는 것을 의미합니다. API 현황 설문조사에서 응답자의 75% 이상이 API 우선 기업의 개발자가 더 행복하고, 신제품을 더 빨리 출시하고, 보안 위험을 더 빨리 제거하고, 더 나은 소프트웨어를 만들고, 생산성이 더 높다는 데 동의했습니다.

또한 API 퍼스트는 비개발자도 앱을 구축할 수 있게 해줍니다. API 현황 보고서에 따르면 API를 사용하는 사람의 약 절반은 비즈니스 분석가, 제품 관리자, CEO와 같은 직군에서 근무하고 있습니다. 이러한 추세는 사용 가능한 서비스와 소프트웨어의 세계를 크게 확장하고 있습니다.

API 플랫폼이 발전함에 따라 코드에 대한 사전 지식이 없는 사람들도 점점 더 일반적인 앱을 구축하고, 테스트와 통합을 실행하고, 데이터를 전송할 수 있게 될 것입니다.
```

```
API first를 위한 다섯 단계
API 우선 기업이 되기 위한 초기 단계는 다음과 같습니다:

- 데이터베이스, 애플리케이션 및 서비스 인벤토리 파악 - 보유하고 있는 API의 수와 API가 부족한 부분을 정확히 파악합니다.
- API를 생성하는 조직의 접근 방식 이해 - 표준 프로세스가 존재하는 곳과 그렇지 않은 곳을 파악합니다.
- 비즈니스 도메인 경계를 정의하고 조직 구조를 이러한 경계에 매핑하세요.
- API 플랫폼을 채택하고 이를 기반으로 표준화하세요.
- 엔지니어링, DevOps 및 제품 관리 팀에 API 우선 교육 실시
```
